## 핵심 요약

"Flipper Zero 공식 펌웨어 개발이 멈췄다"는 커뮤니티의 우려가 확산되자, Flipper Devices가 공식 입장을 내놓았습니다. 결론부터 말하면 **펌웨어 유지보수에 다시 리소스를 배정**하고, 커뮤니티 기여를 받는 방식을 전면 개편한다는 내용입니다.

- 기능 요청은 **GitHub Discussions 투표 기반**으로 우선순위 결정
- Pull Request는 업데이트된 기여 가이드에 따라 **더 엄격하게 심사**
- 모든 펌웨어 변경에 **통합 테스트·회귀 테스트 필수화**, QA 테스트 케이스도 공개
- 개발팀과의 실시간 소통은 종료, **비동기 요청 방식**으로 전환

## Kickstarter의 약속은 이행됐다

Flipper Zero는 크라우드펀딩 시절 "사기 아니냐", "배송 못 할 것"이라는 회의 속에서 출발했습니다. 코로나 이후 부품 부족, 공급망 비용 상승, 수천 건의 비난과 위협까지 겹친 약 1년 반의 압박 끝에, 팀은 다음과 같은 성과로 약속을 지켰다고 자평합니다.

- 모든 후원자에게 기기 배송 완료
- 캠페인에서 공언한 기능 전부 구현
- 소프트웨어 도구, API, SDK를 갖춘 **하드웨어 플랫폼**으로 성장
- 대체 펌웨어·앱·스크립트 등 활발한 커뮤니티 생태계 형성

## 700KB의 제약이 만든 아키텍처 전환

흥미로운 기술적 배경도 있습니다. Flipper Zero의 펌웨어용 플래시 메모리는 **단 700KB**. 새 기능을 계속 넣을 수 없는 한계에 빠르게 부딪혔고, 이를 우회하기 위해 **microSD 카드에서 앱을 동적으로 로드하는 구조**로 전환했습니다.

이 구조 덕분에 핵심 기능 일부를 펌웨어 밖의 앱으로 분리할 수 있었고, Apps Catalog 출시를 거쳐 **2024년 stable firmware 1.0**의 기반이 됐습니다. API와 SDK가 안정화되면서 앱 개발자들이 매달 앱을 다시 빌드해야 하는 고통에서도 벗어났죠. 팀은 "펌웨어에 더 이상 큰 변경이 필요 없다"고 판단하고 새 기기 개발로 무게중심을 옮긴 상태였습니다.

## 달라지는 커뮤니티 기여 방식

사용자가 100만 명을 넘으면서 예전처럼 채팅에 상주하며 실시간으로 소통하는 방식은 불가능해졌습니다. 앞으로는 이렇게 바뀝니다.

### 기능 요청
- **GitHub Discussions**에서만 접수, 사용자 투표로 우선순위 결정
- 팀은 매주 요청을 검토하고 최다 득표 요청을 작업 대상으로 삼음
- 추상적 질문이나 일반 토론은 Discord, Reddit 등 소셜 채널로 분리

### Pull Request
- 업데이트된 기여 가이드에 따라 엄격하게 평가
- 특히 **검증이 어려운 AI 생성 코드**, 저수준 라이브러리 변경, UI·문서에 영향을 주는 변경은 더 까다롭게 심사

### 테스트
- QA 팀이 쓰던 **통합 테스트 케이스를 공개**
- 모든 펌웨어 코드 변경 시 테스트 실행 필수
- 커뮤니티도 회귀 테스트 일부에 참여

## 커뮤니티의 시선은 엇갈린다

Hacker News에서는 반응이 갈렸습니다. "하드웨어는 한 번 팔리는데 펌웨어 업데이트는 영원히 기대받는 구조적 딜레마"라는 지적, "여전히 최소한의 생명 유지 수준으로 들린다"는 회의론이 있었고, 이에 Flipper 개발자가 직접 "개발 목표는 이미 달성됐고, 뜸해진 핵심 펌웨어 업데이트를 해결하려는 것"이라고 답하기도 했습니다.

한편 Momentum, Unleashed 같은 **대체 펌웨어**를 선호하며 공식 펌웨어의 보수적 기능 정책을 비판하는 목소리도 있었는데, 이는 세관·법 집행기관과의 마찰을 피하려는 회사의 책임 회피 전략이라는 해석도 나왔습니다.

## 마치며

"제품이 완성되면 안 되는 이유가 있나?"라는 댓글이 이번 논쟁의 본질을 잘 보여줍니다. 오픈소스 하드웨어 플랫폼이 성숙기에 접어들 때, 회사와 커뮤니티가 지속 가능한 관계를 어떻게 재설계하는지 보여주는 좋은 사례입니다. 투표 기반 기능 요청, AI 생성 코드에 대한 엄격한 심사, 테스트 케이스 공개 같은 방침은 다른 오픈소스 프로젝트에도 참고할 만한 운영 모델입니다.
