## 이런 경험, 다들 한 번쯤 있으시죠?

데스크톱에서 작업하다가 노트북으로 옮기고, 가끔은 서버에서도 개발을 이어가다 보면 이런 순간이 옵니다.

> "어? 이거 예전에 해결했던 문제인데... 그때 어떻게 고쳤더라?"

Claude Code와 Codex는 대화 기록을 각 머신의 `~/.claude/projects`, `~/.codex/sessions`에 JSONL 형태로 차곡차곡 쌓아줍니다. 문제는 파일이 있어도 **검색이 안 된다는 것**. 결국 여러 대의 머신에 SSH로 일일이 접속해서 뒤져야 하는 번거로움이 있었습니다.

이 문제를 해결하기 위해 나온 오픈소스 도구가 바로 **sessionhub**입니다.

## sessionhub는 무엇을 하나요?

핵심은 간단합니다. 흩어진 세션들을 한 곳(허브)에 모아서, 키워드로 검색 가능하게 만들어 줍니다.

```bash
# 키워드로 세션 검색
sessionhub search "connection pool"

# 검색된 세션의 실제 대화 내용 확인
sessionhub raw <id>
```

## 제일 편한 기능은 '에이전트 연동'

sessionhub를 설치하면 Claude Code와 Codex에 스킬이 자동으로 함께 설치됩니다. 그러면 작업 도중 "예전에 이거 어떻게 처리했더라?" 싶은 순간, 에이전트가 알아서 관련 세션을 찾아 컨텍스트로 가져와 줍니다.

```bash
# Claude Code
/sessionhub 저번에 커넥션 풀 터지던거 어떻게 고쳤더라

# Codex
$sessionhub 저번에 커넥션 풀 터지던거 어떻게 고쳤더라
```

따로 검색 결과를 복사해서 붙여넣을 필요 없이, 자연스럽게 이전 작업 맥락을 대화에 이어붙일 수 있는 거죠.

## 동작 방식: 가볍고 효율적으로

- 여러 머신 중 **허브 하나를 지정**해두면, 나머지 머신의 세션이 15분마다 SSH로 허브에 동기화됩니다.
- 허브가 아닌 다른 머신에서 검색을 요청하면, 허브가 대신 찾아서 결과를 돌려줍니다.
- 무거운 로그 파일을 여기저기 복사할 필요가 없습니다.

특히 인상적인 부분은 **용량 최적화**입니다. 개발자가 직접 로그를 분석해보니, 저장된 데이터의 99.7%는 tool 호출·스트리밍 프레임·토큰 계산 같은 '기계용' 데이터였고, 실제로 다시 찾아보게 되는 대화 내용은 0.3%에 불과했다고 합니다. 그래서 대화 부분만 뽑아 압축해 인덱스에 저장하는 방식을 택했고, 그 결과 원본 약 10GB가 수백 MB 수준으로 줄었다고 하네요. 원본 전체는 각 머신에 그대로 남아있으니, 필요하면 언제든 열어볼 수 있습니다.

## 설치는 이렇게

```bash
uv tool install git+https://github.com/esc5221/sessionhub
sessionhub setup
```

`setup` 명령 하나로 허브 설정과 에이전트 스킬 설치까지 안내해줍니다.

## 왜 '메모리'가 아니라 '검색'을 택했을까?

요즘 AI 에이전트에 장기 메모리를 붙이는 시도가 많습니다. 지난 대화를 요약·추출해서 별도 메모리에 쌓아두고, 다음 세션이 참고하게 하는 방식이죠. 개발자는 이 접근도 좋아하지만 두 가지 아쉬운 점을 꼽았습니다.

1. 요약하는 순간 원래 맥락이 조금씩 손실된다.
2. 쌓인 메모리는 시간이 지나면 낡아져서(stale) 계속 손봐줘야 한다.

그래서 sessionhub는 별도의 요약본을 만들지 않고, **원본 대화를 그대로 둔 채 필요할 때마다 다시 꺼내보는 방식**을 택했습니다. 굳이 분류하자면 memory-centric보다는 **compute-centric**에 가까운 철학이라고 할 수 있습니다.

## 마무리

여러 기기를 오가며 AI 코딩 에이전트를 쓰는 개발자라면, "그때 그 대화 어디 있었지?" 하는 답답함을 한 번쯤 겪어봤을 겁니다. sessionhub는 이 문제를 거창한 인프라 없이, SSH 동기화와 가벼운 인덱싱만으로 실용적으로 풀어낸 사례로 보입니다. 과거 컨텍스트를 에이전트에게 낡지 않게, 손실 없이 전달하는 방법에 대해 고민하고 있다면 한 번 살펴볼 만한 도구입니다.
