## 한눈에 보는 요약

"아직도 Emacs를 쓰는 사람이 있나요?" 한 개발자가 던진 이 질문에 대한 답은, 그의 25년 편집기 여정 그 자체입니다. 1997년 Linux 입문과 함께 Vim·Emacs를 오갔고, 2015년 VSCode와 IntelliJ로 갈아탔다가, 2022년 다시 **Doom Emacs**로 돌아왔습니다. 화려한 IDE를 모두 경험한 뒤에도 Emacs로 회귀한 이유는 의외로 단순합니다. **어떤 머신에서든 똑같은 개발 환경을 유지할 수 있다는 것**이죠.

## 편집기 여정: IDE에서 Linux 편집기로, 다시 IDE로

글쓴이의 편집기 역사는 개발 도구의 흐름을 그대로 보여줍니다.

- **1997년 ~ 2015년**: Caldera OpenLinux로 Linux에 입문하며 Vim과 Emacs를 익힘. 긴 코딩 세션엔 Emacs, 여러 파일을 빠르게 오가는 작업엔 Vim을 사용.
- **2015년 ~ 2022년**: 언어 통합이 약한 점이 아쉬워 VSCode로 이동. 현대적인 UI, 가벼운 규모, **JSON 기반 설정**, 그리고 Go·Rust의 **LSP 통합**이 결정적이었음. Java 프로젝트에서는 IntelliJ가 더 현실적인 선택지.
- **2022년 이후**: Snowflake로 옮긴 뒤 다시 Emacs로 회귀.

## 왜 다시 Emacs였을까

Snowflake에서의 업무는 오래된 Linux VM 안에서 **shell script와 Bazel build file**을 작성하는 일이 중심이었습니다. 이 환경에서는 VSCode나 IntelliJ가 강점을 살리지 못했고, 원격 그래픽 환경의 제약도 거슬렸습니다. 결국 SSH로 접속해 작업하는 흐름으로 돌아갔고, 긴 작업 세션에 어울리는 편집기가 다시 필요해졌습니다. 그 답이 Emacs였죠.

다만 수년간 쌓여 스스로도 이해하지 못하게 된 수백 줄짜리 `init.el`은 과감히 버리고, 새로 시작하기로 합니다. 이때 만난 것이 **Doom Emacs**입니다.

## Doom Emacs가 바꾼 경험

Doom Emacs는 Emacs를 처음부터 구성해 둔 일종의 **"배포판"**입니다. IDE라고 주장하진 않지만, 실제 사용감은 IDE에 가깝습니다.

- **합리적인 기본값**과 미리 정의된 언어 통합 제공
- **Vim 스타일 키 바인딩**으로 ex-Vim 사용자에게 친숙
- `SPC`(스페이스) 기반 팝업 메뉴 — 잠깐 멈추면 가능한 명령을 설명하는 메뉴가 떠 **기능 발견이 쉬워짐**
- LSP와 tree-sitter 같은 현대 기능 덕분에 대부분의 언어에서 제대로 된 통합을 얻음

설정도 세 파일로 단순하게 나뉩니다.

- `config.el`: theme, font 같은 전역 설정
- `init.el`: 활성화할 Doom 모듈 선택
- `packages.el`: Doom에 없는 패키지 설치

## 진짜 핵심: "어디서든 같은 환경"

글쓴이가 꼽는 가장 강력한 가치는 화려한 기능이 아닙니다. MacBook, Linux laptop, 클라우드 워크스테이션, 개인 FreeBSD 서버까지 — 어떤 머신에서 일하든 **shell, tmux, Emacs만 있으면 동일한 환경과 근육 기억**을 그대로 가져갈 수 있다는 점입니다. 여러 환경을 오가는 개발자에게 이 일관성은 생산성으로 직결됩니다.

## 커뮤니티의 반응

Lobste.rs 토론에서도 흥미로운 관점이 오갔습니다.

- **"고대 유물"이 다시 주목받는다**: CLI 코딩 에이전트가 늘면서 tmux, Vim, Emacs 같은 도구들이 재발견되고 있다는 의견. 오래 살아남은 데는 이유가 있었던 셈입니다.
- **키 바인딩의 진입 장벽**: 2026년에 Emacs를 새로 배우려면 수십 년 쌓인 암묵지에 압도되기 쉽다는 토로. 기본 키 바인딩이 낯설다면 Vim 사용자를 위한 **evil**, 혹은 **meow·boon** 같은 모달 편집 패키지를 권하는 조언이 이어졌습니다.
- **"Emacs의 참된 복음"**: 한 베테랑 사용자는 "자신에게 가장 잘 맞는 방식으로 쓰는 것이야말로 진짜 정답"이라며, evil을 써도 죄책감을 가질 필요가 없다고 말합니다. "Emacs가 사용자를 바꾸기도 하지만, 그보다 훨씬 크게 Emacs가 사용자를 위해 바뀌어준다"는 표현이 인상적입니다.

## 마치며

Doom Emacs가 "너무 많은 일을 한다"는 불만도 있지만, 글쓴이는 "실제로 많은 일을 해주기 때문에 유용하다"고 답합니다. 결국 이 글은 "무엇이 가장 좋은 편집기인가"라는 질문이 아니라, **자신의 작업 흐름에 가장 잘 맞는 도구를 찾아가는 과정**에 대한 이야기입니다. 화려한 IDE를 모두 거친 개발자가 다시 텍스트 기반 도구로 돌아왔다는 사실은, 오래된 도구가 여전히 유효한 이유를 새삼 곱씹게 합니다.
